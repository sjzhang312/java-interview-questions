## 1、ConcurrentHashMap和Hashtable的区别

底层数据结构：

- JDK1.7中的ConcurrentHashMap底层采用分段的数组+链表实现，JDK1.8采用的数据结构跟HashMap1.8的结构一样，数组+链表/红黑二叉树。
- Hashtable和Jdk1.8之前的HashMap的底层数据结构类似都是采用数组+链表的形式，数组是HashMap的，链表是为了解决哈希冲突而存在的。

实现线程安全的方式（重要）：

- 在Jdk1.7的时候，ConcurrentHashMap对整个桶数组进行了分割分段（Segment，分段锁），每一把锁只锁容器其中一部分数据多线程访问容器里不同数据段的数据，就不会存在锁竞争，提高并发效率。、
- 到了JDK1.8的时候，ConcueentHashMap已经摒弃了segment概念，而是直接采用Node数组+链表+红黑树的数据结构来实现，并发控制使用synchronized和CAS来操作。(头结点加锁)
- Hashtable（同一把锁）：使用sybchrozied来保证线程安全，效率非常低下。当一个线程访问同步方法时，其他线程也访问同步方法可能会进入阻塞或轮询状态，如使用put添加元素，另一个线程不能使用put添加元素，也不能使用get，竞争会越来越激烈。

## 2、java中如何创建一个线程

在Java中，创建线程有以下几种主要方式，以下是详细说明和代码示例：

### 1) **继承 `Thread` 类**

- 创建一个类，继承`Thread`类，并重写`run()`方法。
- 创建该类的实例并调用`start()`方法启动线程。  

```java
public class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Thread is running: " + Thread.currentThread().getName());
    }

    public static void main(String[] args) {
        MyThread thread = new MyThread();
        thread.start(); // 启动线程
    }
}
```

**说明**：

- `run()`方法定义线程的执行逻辑。
- 调用`start()`启动线程，JVM会调用`run()`方法。
- 直接调用`run()`不会创建新线程，只是在当前线程中执行`run()`方法。

---

### 2) **实现 `Runnable` 接口**

- 创建一个类，实现`Runnable`接口，并实现`run()`方法。
- 将`Runnable`对象传递给`Thread`构造器，创建线程并启动。

```java
public class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println("Thread is running: " + Thread.currentThread().getName());
    }

    public static void main(String[] args) {
        Runnable runnable = new MyRunnable();
        Thread thread = new Thread(runnable);
        thread.start(); // 启动线程
    }
}
```

**说明**：

- `Runnable`接口更灵活，因为Java只支持单继承，实现了`Runnable`的类还可以继承其他类。
- 推荐使用`Runnable`而不是直接继承`Thread`。

---

### 3)**使用 Lambda 表达式（Java 8+）**

- 由于`Runnable`是函数式接口，可以用Lambda表达式简化代码。
- 直接将Lambda表达式传递给`Thread`构造器。

```java
public class LambdaThread {
    public static void main(String[] args) {
        Thread thread = new Thread(() -> {
            System.out.println("Thread is running: " + Thread.currentThread().getName());
        });
        thread.start(); // 启动线程
    }
}
```

**说明**：

- 代码更简洁，适合简单的线程任务。
- Lambda表达式替代了显式的`Runnable`实现。

**`->` 就是 Lambda 的 “箭头”**，作用是：

**把左边的参数 → 传给右边的代码执行**

---

### 4) 使用 `ExecutorService`（线程池）

- 通过`ExecutorService`创建线程池，提交任务来执行线程。
- 适合需要管理多个线程的场景，避免手动创建大量线程。

```java
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class ExecutorThread {
    public static void main(String[] args) {
        ExecutorService executor = Executors.newFixedThreadPool(2); // 创建线程池
        executor.submit(() -> {
            System.out.println("Thread is running: " + Thread.currentThread().getName());
        });
        executor.shutdown(); // 关闭线程池
    }
}
```

**说明**：

- `Executors.newFixedThreadPool(n)`创建固定大小的线程池。
- 使用`submit()`提交`Runnable`或`Callable`任务。
- 需要调用`shutdown()`关闭线程池，否则程序可能不会终止。

---

### 5) 实现 `Callable` 接口（带返回值）

- 创建一个类，实现`Callable`接口，并实现`call()`方法。
- 使用`ExecutorService`提交`Callable`任务，获取`Future`对象以处理返回值。

```java
import java.util.concurrent.*;

public class MyCallable implements Callable<String> {
    @Override
    public String call() {
        return "Thread is running: " + Thread.currentThread().getName();
    }

    public static void main(String[] args) throws Exception {
        ExecutorService executor = Executors.newFixedThreadPool(1);
        Future<String> future = executor.submit(new MyCallable());
        System.out.println(future.get()); // 获取线程执行结果
        executor.shutdown();
    }
}
```

## 3、hashmap在1.8之前和之后有什么区别？

1）底层数据结构的变化

1.8之前：HashMap的底层实现是数组+链表，当发生哈希冲突时，会将这些键值对以链表的形式存储到对应的数组槽中

问题：如果哈西冲突严重，链表会变得很长导致查找、插入、删除的时间复杂度从O(1)变成O(n)

2）1.8之后：

- 引入了红黑树来优化链表过长的问题。当某个桶（数组槽）中的链表长度超过一定的阈值（默认是8），并且HashMap容量达到一定大小（默认是64），链表会转为红黑树
- 红黑树的查找、插入和删除的时间复杂度是O（logn）

2）哈希冲突的优化处理

Jdk1.8之前  ：    链表插入新元素采用头插法

1.8之后，    链表插入元素改为尾插法

3）扩容时的性能优化

Jdk1.8之前：

在扩容时，需要重新计算所有元素的哈希值并重新分配到新的数组中，开销较大

Jdk1,8之后

扩容时利用了位运算的优化，扩容后，元素要么留在原位置，要么移动原位置+旧容量的新位置。这种优化减少了重新计算哈希的次数，提高了扩容效率。

## 12、单核CPU支持Java多线程吗

单核CPU是支持Java多线程的。操作系统通过时间片轮转的方式，将CPU的时间分配给不同的线程。尽管单核CPU一次只能执行一个任务，但通过快速在多个线程之间转换，可以让用户感觉多个任务是同时进行的。

## 13、单核CPU上运行多个线程效率一定会高吗？

单核CPU同时运行多个线程的效率是否为空，取决于线程的类型和任务的性质。一般来说，有两种类型的线程：

- CPU密集型：CPU密集型的线程主要进行计算和逻辑处理，需要占用大量的CPU资源
- IO密集型：IO密集型的线程主要进行输入输出操作，如读写文件，网络通信等，需要等待IO设备的响应，而不占用太多的CPU资源

在单核CPU上，同一时刻只能有一个线程在执行，其他线程需要等待CPU的时间片分配。如果线程是CPU密集型的，那么多个线程同时运行会导致频繁的线程切换，增加了系统的开销，降低了效率。如果线程是IO密集型的，那么多个线程同时运行可以利用CPU在等待IO时的空闲时间，提高了效率。

## 14、java中参数传递机制

- 基本数据类型（如int）：按值传递，方法接受的是值的副本，对副本的修改不会影响原始变量
- 对象引用：按引用传递，方法接收的是对象引用的副本。可以通过引用修改对象的状态，但重新赋值给引用本身不会影响原始引用
- 数组（如int[]）：也是按引用传递，方法接受的数组引用的副本，可以通过引用修改数组内容。

## 22、介绍一下雪花算法

雪花算法是一种分布式唯一ID生成算法，用于在分布式系统中生成全局唯一、趋势递增的64位ID。

它的ID由64位long类型组成，一般结构是

1位符号位+41位时间戳+10位机器标识+12位序列号

- 时间戳确保整体有序递增
- 机器标识保证节点之间不会冲突
- 序列号解决同一毫秒内生成多个ID的问题

缺点：

1.依赖系统时钟，若服务器时钟回拨会导致ID冲突

## 24、sleep会释放CPU吗

是的。调用Thread.sleep()时，线程会释放CPU，但不会释放持有的锁。

当线程调用sleep()后，会主动让出CPU时间片，进入TIMED_WAITING状态。此时操作系统会触发调度，将CPU分配给处于就绪状态的线程。这样其他线程便有机会执行。

Sleep（）不会释放线程已持有的任何锁（如synchronized同步代码块或方法中获取的锁）。因此，如果有其他线程获取同一把锁，他们仍让会被阻塞，直到原线程退出同步代码块。

- 使用 AQS 的 tryAcquire 和 tryRelease 方法实现锁的获取和释放。
- 公平锁（FairSync）和非公平锁（NonfairSync）分别通过不同的 tryAcquire 实现来控制线程获取锁的顺序。

**2）ReentrantReadWriteLock**

描述：ReentrantReadWriteLock 是一个读写锁，支持多个线程同时读取，但写操作互斥，且支持重入。

**3) Semaphore**

描述：Semaphore 是一个计数信号量，用于控制同时访问某个资源的线程数量

**4）CountDownLatch**

- 描述：CountDownLatch 是一个倒计数锁，允许线程等待直到计数器减为 0。
- AQS 使用
- 内部通过 CountDownLatch.Sync（继承自 AQS）实现
- AQS 的 state 字段表示计数器的当前值。
- await 方法调用 tryAcquireShared，检查 state 是否为 0，若不为 0 则线程等待
- countDown 方法调用 tryReleaseShared，将 state 减 1，当 state 达到 0 时唤醒所有等待线程

## 7、ArrayList扩容机制

1. **第一步：初始化（容量为0）**
   当你使用无参构造器 new ArrayList<>() 时，底层并没有马上分配存储空间。它内部使用一个名为 DEFAULTCAPACITY_EMPTY_ELEMENTDATA 的空数组（容量为0）进行初始化，这是一种懒加载策略，为了节省内存。

2. **第二步：添加第1个元素（容量 0 → 10）**
   当你第一次调用 add() 方法添加元素时，扩容机制被触发：

   检查：代码发现当前数组是空数组。

计算：它会取“默认容量10”和“所需最小容量1”之间的最大值，因此最小需要容量被定为10。

执行：因为旧容量0小于10，进入 grow() 方法。虽然按1.5倍算法计算（0 + 0/2）结果还是0，但代码有一个保底逻辑：如果计算出的新容量小于最小需要容量（10），就直接使用最小需要容量。因此，数组一次性被扩容为容量10的新数组。

3. **第三步：添加第2~10个元素（无需扩容）**
   此时数组容量为10。当你陆续添加第2个到第10个元素时，每次检查都会发现 size + 1（最多为10）仍然小于等于当前的数组长度（10），因此不会触发扩容，只是按顺序将元素放入数组中。

4. **第四步：添加第11个元素（容量 10 → 15，1.5倍）**
   当你尝试添加第11个元素时，容量再次不足：

触发：所需最小容量为11，当前容量10，需要扩容。

计算：进入 grow(11) 方法。

旧容量 oldCapacity = 10

新容量 newCapacity = oldCapacity + (oldCapacity >> 1) = 10 + 5 = 15。

>> 1 是右移一位的位运算，效率很高，等价于除以2。

**这是扩容大小（容量），具体怎么扩容的？**

1. 创建新的数组，根据计算得到的新容量
2. 将原数组的元素复制到新数组当中
3. 更新引用，将ArrayList<>指向为新数组

## 9、AQS是什么？

AQS是AbstractQueeuedSynchronizer的简称，即抽象队列同步器

- 队列：使用先进先出（FIFO）的队列存储数据
- 同步：实现了同步功能

那AQS有什么用呢？

AQS是一个用来构建锁和同步器的框架，使用AQS能简单且高效地构造出应用广泛的同步器，比如ReentrantLock ， Semaphore ，ReentrantReadWriteLock， FutureTask等等，都是基于AQS的

## 10、说说你对原子性、可见性、有序性的理解

**原子性**要求一个操作是不可分割的，要么全部执行成功，要么完全不执行。

**可见性**要求一个线程对共享变量的修改，能够立即被其他线程即时看见。

**有序性**是指程序执行的顺序是否按照代码编写的顺序执行。

## 11、讲下指令重排？

指令重排是值CPU或编译器为了提高程序的执行效率，改变代码执行顺序的一种优化技术。

指令重排可能会导致双重检查锁失效，比如下面的单例模式代码

```java
public class Singleton {
    private static Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) { // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) { // 第二次检查
                    instance = new Singleton(); // 可能发生指令重排
                }
            }
        }
        return instance;
    }
}
```

如果线程 A 执行了 instance = new Singleton();，但构造方法还没执行完，线程 B 可能会读取到一个未初始化的对象，导致出现空指针异常。

## 12、怎么保证线程安全

线程安全 是指在并发环境下，多个线程访问共享资源时，程序能够正确地执行，而不会出现数据不一致的问题



为了保证线程安全，可以使用synchronized关键字 对方法加锁，对代码块加锁。线程在执行同步方法、同步代码快时，会获得类锁或者对象锁，其他线程就会阻塞并等待锁。

## 1、对象什么时候会进入老年代？

对象通常会在年轻代中分配，随着时间的推移和垃圾收集的进程，某些满足条件的对象会进入到老年代中，如长期存活的对象。

- **长期存活的对象**
- **大对象**
- **动态年龄判断**

1）长期存活的对象如何判断？

​	JVM会给对象维护一个“年龄”计数器，记录对象在新生代中经历Minor GC的次数。每次GC未被回收的对象，其年龄加1。

当超过一个特定的阈值，默认值15，就会被认为老对象了，需要重点关照。

2）动态年龄判断

​	如果 Survivor 区中**所有同龄对象的总大小**大于或等于 **Survivor 区的一半**，那么年龄**大于等于**该年龄的对象就可以直接进入老年代，无需等到阈值上限

3）大对象如何判断？

​	大对象是指占用**内存较大的对象，如大数组、长字符串等。**

```java
Int [] array =new int[];
String str= new String( new char[1000000]);

```
## 2、垃圾回收的过程是什么？

Java的垃圾回收主要分为标记存活对象、清楚无用对象、以及内存压缩/整理三个阶段。不同的垃圾回收器在执行这些步骤时会采用不同的策略和算法

Java通过可达性分析算法来判断一个对象是否还存活。

通过一组名为GC Roots 的根对象，进行递归扫描，无法从根对象到达的对象就是垃圾，可以被回收。（这也是G1、CMS等主流垃圾收集器使用的主要算法）

## 3、垃圾回收算法了解吗

分代收集算法是目前主流的垃圾回收算法，它根据对象存活周期的不同将内存划分为几块，一般分为新生代和老年代。

![](images/image15.png)

新生代用**复制**算法，因为大部分对象生命周期短。老年代用**标记-清除** / **标记-整理**算法，因为对象存活率较高。

**G1还属于分类吗？**

Garbage-First (G1) Garbage Collector 仍然是分代垃圾收集器，但它不再使用物理上连续的新生代和老年代，而是通过 Region 逻辑划分实现分代。

Heap

 └─ Old Generation

**G1的结构**    

G1把整个堆拆分成**很多Region**，没有固定的 Young / Old 物理边界但是逻辑上仍然存在分代

Heap
 ├─ Region
 ├─ Region
 ├─ Region
 ├─ Region
 ├─ Region

每个 Region 可以被标记为Edan 、 Survivor、Old、Humongous



## 4、知道哪些垃圾回收器

JVM的垃圾回收器主要分为两大类：分代收集器和分区收集器，**分代收集器的代表是CMS，分区收集器的代码是G1和ZGC。**

说说CMS收集器？

**CMS**在JDK1.5时引入，JDK9 被标记弃用，JDK14时被移除

CMS时一种低延迟的垃圾收集器，采用标记清除算法、分为初始标记、并发标记、重新标记和并发清除四个阶段，优点是 垃圾回收线程和应用线程 同时进行，停顿时间短，适合延迟敏感的应用，但容易产生内存碎片，可能触发Full GC

说说G1收集器？

**G1**在JDK1.7时引入，在JDK9时取代CMS成为默认的垃圾回收器。

G1是一种面向大内存、高吞吐量的垃圾收集器，它将堆划分为多个小的Region ，通过标记整理算法，避免了内存碎片问题。优点是停顿时间可控，适合大堆场景，但调优困难。

## 6、GC Roots的引用有哪几种？

## 7、Minor GC、Major GC、Mixed GC、Full GC 区别？

Minor gc也称为Young GC，是指发生在年轻代的垃圾收集。年轻代包含Eden区以及两个Survivor区。

![](images/image17.png)

Major GC也称为Old GC，主要指的是发生在老年代的垃圾收集。是CMS特有行为

Mixed GC是G1垃圾收集器特有的一种GC类型，它在一次GC中同时清理年轻代和部分老年代

Full GC是最彻底的垃圾收集，涉及整个java堆和方法区。它是最耗时的GC，通常在JVM压力很大时发生。

Full GC是怎么去清理的？

Full GC会从GC Root 出发，标记所有可达对象。新生代使用复制算法，清空Eden区。老年代使用标记整理算法，回收对象并消除碎片。

停顿时间较长，会影响系统响应性能。

## 8、新生代频繁minor gc怎么办？

频繁的Minor GC通常意味着新生代中的对象频繁地被垃圾回收，可能是因为新生代空间设置的过小，或者是因为程序中存在大量的短生命周期对象（如临时变量）

## 9、频繁Full GC怎么办？

频繁的Full GC通常意味着老年代中的对象频繁地被垃圾回收：

1）可能是因为老年代空间设置的过小

2）或者因为程序中存在大量的长生命周期对象

## 10、线程池的工作队列是如何实现阻塞的？

线程池的工作队列通常是使用BlockingQueue接口的实现类，如ArrayBlockingQueue或LinkedBlockingQueue。这些队列通过内部的ReentrantLock和Condition实现了阻塞机：当队列满时，调用put（）的线程会被阻塞；当队列为空时，调用take()的线程会阻塞，直到有可用元素或空间。

## 11、介绍一下CMS （JDK8）

CMS是一种老年代垃圾回收器，其主要目标是获取最短垃圾回收停顿时间，它使用的是多线程的标记-清除算法。

CMS工作机制分为以下4个阶段：

- 初始阶段：
- 并发阶段
- 重新标记
- 并发清除

由于耗时最长的并发标记和并发清除过程中，垃圾收集线程可以和用户现在一起并发工作，所以总体上来看CMS 收集器的内存回收和用户线程是一起并发地执行。



## 12、介绍一下G1（JDK9）

是⼀款⾯向服务器的垃圾收集器,主要针对配备多颗处理器及⼤容量内存的机器.以极⾼概率满⾜GC停顿时间要求的同时,还具备⾼吞吐量性能特征；相比与 CMS 收集器，G1 收集器两个最突出的改进是：

- 基于标记-整理算法，不产生内存碎片
- 可以非常精确控制停顿时间，在不牺牲吞吐量前提下，实现低停顿垃圾回收。

G1 收集器避免全区域垃圾收集，它把堆内存划分为大小固定的几个独立区域，并且跟踪这些区域的垃圾收集进度，同时在后台维护一个优先级列表，每次根据所允许的收集时间，优先回收垃圾最多的区域。区域划分和优先级区域回收机制，确保 G1 收集器可以在有限时间获得最高的垃圾收集效率。

- 初始标记：Stop The World，仅使用一条初始标记线程对GC Roots关联的对象进行标记；

- 并发标记：使用一条标记线程与用户线程并发执行。此过程进行可达性分析，速度很慢；

- 混合标记：回收垃圾最多的一块Region   优点之停止局部stw， 提高了性能

- 筛选回收：回收废弃对象，此时也要 Stop The World，并使用多条筛选回收线程并发执行；

## 11、为什么Idea不推荐使用@Autowired注解注入Bean？

1）第一个字段注入不利于单元测试。字段注入需要使用反射或Spring容器才能注入依赖，测试更加复杂；而构造方法可以直接通过传入Mock对象，测试起来更简单

2）第二个字段注入会隐藏循环依赖问题，而构造方法注入会在项目启动时就去检查依赖关系，能更早的发现问题

3）构造方法注入可以使用final字段确保依赖在对象创建时就会被初始化，避免了后续修改的风险

@Autowired和@Resource注解的区别？

@Autowired时Spring框架提供的注解，而@Resource时Java EE标准提供的注解，换句话说，@Resource时JDK自带的，而@Autowired时Spring特有的

从注入方式说，@Autowired默认按照类型，也就是byType进行注入，而@Resource默认按照名称，也就是byName进行注入。

## 12、@Autowired的实现原理了解吗？

@Autowired是Spring实现依赖注入的核心注解，其实现原理基于反射机制和BeanPostProcessor接口。

整个过程为两个主要阶段。第一个阶段是依赖收集阶段，发生在Bean实例化之后，属性赋值之前。Autowired的Processor会扫描Bean的所有字段，方法和构造方法，找出标注了@Autowired注解的地方，然后把这些信息封装成Injection元数据对象缓存起来。这个过程用到了大量的反射操作，需要分析类的结构、注解信息等等。

第二个阶段是依赖注入阶段，Spring会取出之前缓存的Injection元数据对象，然后逐个处理每个注入点。对于每个@Autowired标注的字段或方法，Spring会根据类型去容器中查找匹配的Bean。

## 13、Spring事务的理解

Spring提供了两种事务管理方式，

**编程式事务**和**声明式事务。**

编程式事务就是用我们手动调用事务的开始、提交、回滚等操作，虽然灵活但是代码比较繁琐。

声明式事务只要在事务的方法上加上@Transactional注解就行，Spring事务会帮我们自动处理事务的整个生命周期。

编程式事务可以使用TransactionTemplate和PlatformTransactionManager来实现，允许我们直接控制事务的边界。

Spring事务的底层实现是通过AOP来完成的。当我们在方法加上@Transactional注解后，Spring会为这个Bean创建代理对象，在方法执行前开启事务，方法正常返回提交事务，方法抛出异常就回滚事务。 （声明式事务的优点是不需要在业务逻辑中掺杂事务管理的代码，缺点是最细粒度只能到方法级别，无法到代码块级别）



## 14、声明式事务的实现原理了解吗？

Spring的声明式事务管理是通过AOP和代理机制实现的，大致可以分为两个阶段。

第一个阶段发生在Spring容器启动时，它会扫描所有Bean。如果发现某个Bean的方法上标注了@Transactional注解，Spring不会直接返回这个原始Bean的实例。而是为这个Bean创建一个代理对象。这个代理对象拥有和原始对象完全相同的方法，但在内部悄悄包裹了事务处理的逻辑。

第二个阶段发生在方法调用的运行阶段，当我们的代码调用那个被@Transactional注解修饰的方法时，实际上调用的是Spring创建的那个代理对象的方法。

1）事务拦截器会在代理对象执行真正的业务逻辑之前，根据@Transactional注解的配置获取事务属性，比如传播行为、隔离级别等，

2）然后通过事务管理器来开启一个新的事务。并从数据库连接池获取一个连接，关闭其自动提交。

3）接着，代理对象会调用原始Bean实例中真正的业务方法，

A）如果业务方法顺利执行完毕，没有抛出任何异常，那么拦截器就会通过事务管理器提交事务，将之前的所有数据库操作保存。

B）如果业务方法抛出了异常，拦截器会捕获到这个异常，并通过事务管理器回滚事务，将之前的所有数据库操作撤销。

4）最后，无论事务是提交还是回滚，拦截器都会释放数据库连接。

## 15、说说事务的传播机制？

简单来讲，当一个事务方法A调用另一个事务方法B时，方法B的事务应该如何运行？是加入方法A的现有事务，还是开启一个新的事务，或者以非事务方式运行？这就是事务传播机制要解决的问题。

Spring定义了七种事务传播的行为，其中REQUIRED是默认的传播行为，表示如果当前存在事务，则加入该事务；如果当前没有事务，则创建一个新的事务。

## 18、如何实现动态代理

动态代理是一种非常强大的设计功能，它允许我们在不修改源代码的情况下，对一个类或对象的方法进行功能增强。

在java中，实现动态代理最主流的方式有两种，JDK动态代理和CGLIB动态代理。

第一种：JDK动态代理

Java官方提供的，其核心要求是目标类必须实现一个或多个接口，JDK动态代理在运行时，会利用Proxy.newProxyInstance()方法，动态地创建一个实现了这些接口的代理类的实例。这个代理类在内存中生成，你看不到它的.java或.class文件。

当你调用对象的任何一个方法时，这个调用都会被转发到我们提供的一个InvocationHandler接口的invoke方法中。在invoke方法里，我们就可以调用原始方法（目标方法）之前或之后，加入我们自己的增强逻辑。

第二种，CGLIB动态代理

CGLIB是一个第三方的代码生成库。它的原理与JDK完全不同，它不要求被代理的类实现接口，它在运行时，动态生成目标类的子类作为代理类（通过ASM字节码操作技术）。然后，它会重写父类（也就是被代理类）中所有非final，private和static的方法。

当你调用代理对象的任何一个方法时，这个调用会被CGLIB的methodInterceptor接口的intercept方法拦截。和InvocationHandler的invoke方法一样，我们可以在intercept方法里，在调用原始的父类方法之前或之后，加入我们的增强逻辑。

## 19、SPI是什么？和API有什么区别？

SPI即Service Provider Interface，字面意思就是“服务提供者的接口”，我的理解是：专门提供给服务提供者或者拓展框架功能的开发者去使用的一个接口。

SPI将服务接口和具体的服务实现分离开来，将服务调用方和服务实现者解耦，能够提升程序的拓展性、可维护性。修改或者替换服务实现并不需要修改调用方。

很多框架都使用了java的SPI机制，比如：Spring框架、数据库加载驱动、日志接口、以及Dubbo的拓展实现等等。

# Redis

## 1、Redis为什么快呢？

基于内存的数据存储，Redis将数据存储在内存中，使得数据的读写避开了磁盘IO

单线程模型

IO多路复用，基于Linux’的select/epoll机制。该机制允许内核中同时存在多个监听套接字和已连接套接字，内核会一直监听这些套接字的连接请求 或者数据请求，一旦有请求到达，就会交给Redis处理，就实现了所谓的Reids单个线程处理多个IO读写的请求。

Select和epoll是Linux提供的IO多路复用机制，允许一个进程同时监控多个文件描述符（例如套接字），以检查它们是否可读、可写或发生错误。具体来说：

Select通过轮询检查一组文件描述符的状态，判断哪些描述符有事件*如数据库到达或连接请求

Epoll：更高效的事件驱动机制，基于内核事件表，仅通知有事件发生（例如新连接或数据到达）的文件描述符，减少轮询开销

在Redis中，单个线程利用epoll（或select，视系统配置）监控多个客户端套接字（包括监听套接字和已连接套接字）。当某个套接字有事件（如新连接请求或数据到达）时，内核会通知Redis，Redis再处理这些事件。这种方法让Redis单线程能高效处理多个并发IO请求。

2）内核一直监听，内核不干别的了？

内核的职责：Linux内核是操作系统核心，负责管理硬件资源（如CPU、内存、IO设置）、调度进程、处理中断。监听套接字只是内核网络子系统的一部分工作，具体由内核的网络协议栈和IO管理模块处理。

事件驱动机制在epoll中，内核不是“忙等”（即不断轮询检查套接字状态）。相反，内核维护一个事件表，当网卡接受到数据或有新连接请求时，会触发硬件中断，内核的网络协议栈处理这些中断，将事件加入epoll事件队列。Redis通过系统调用（如epoll_wait）获取这些事件，处理后再返回控制权给内核。

非阻塞处理；内核处理这些事件是异步的，不会一直占用CPU。内核在处理完网络事件后，会继续执行其他任务（如调度其他进程、处理文件IO）等

3）内核是CPU吗？

内核：内核是操作系统的核心软件，运行在CPU中，负责管理硬件资源、提供系统调用接口给用户程序。它是一段代码，存储在内存中，由CPU执行。

CPU是硬件，负责执行指令，包括内核代码和用户程序代码。内核通过CPU来完成任务

## 6、Redis中高可用性

## 7、了解Redis Cluster（集群）吗

Redis Cluster是Redis通过数据分片（sharding）将数据分散到多个节点，支持高并发读写、自动故障转移和动态扩展。

1）数据分片（sharding）

- Redis Cluster将数据分成16384个槽（hash slots），每个槽存储一部分数据。
- 键（key）通过CRC16(key) %16384 算法映射到某个槽
- 每个节点负责一部分槽，数据根据槽分布到不同的节点上
- 客户端通过键计算槽位置，访问对应的节点

2）节点角色

- 主节点（Master）：负责读写操作，每个主节点管理一部分槽
- 从节点（Slave）：为主节点提供数据复制和高可用性，通常只读，故障时可升级为主节点
- 集群中通常有多个主节点，每个主节点可以有0或多个从节点

## 8、Redis怎么处理大Key？

大Key 是指占用内存空间较大的缓存键，比如超过10M的键值对。常见的大Key类型包括：包含大量元素的List、Set、Hash结构 ，存储大文件的String类型，以及包含复杂嵌套对象的JSON数据等。

对于大Key问题，最根本的解决方案是拆分大Key，将其拆分为多个小Key存储。比如将一个包含大量用户信息的Hash拆分成多个小Hash

## 9、Redis6.0之后采用的单Reactor多线程IO模型流程

1）主线程接受连接

- 主线程运行事件循环，基于NIO多路复用（如epoll）监听新连接（accept）和socket事件
- 当有新客户端连接时，主线程接受连接并分配socket给某个IO线程

2)主线程分派socket事件

主线程通过多路复用检测到 socket 上的事件（例如可读或可写），将这些事件分派给某个 I/O 线程处理。

3)IO线程读取和解析

- 分配的UI线程从客户端socket读取请数据(Redis协议格式的字节流)
- IO线程解析这些字节流，转换为Redis内部的命令结构

4）主线程执行命令

I/O 线程将解析好的命令交给主线程，主线程在内存中执行命令（例如 SET key value或 GET key），操作 Redis 的数据结构。

5）I/O线程返回响应

- 主线程执行完命令后生成响应，将结果交给同一个IO线程
- IO线程通过非阻塞写操作将响应写回客户端。

## 6、有了binlog为什么还要undolog redolog?

Binlog属于Server层，与存储引擎无关，无法直接操作物理数据页。而redo log和undo log是InnoDB存储引擎实现ACID的基石。

Binlog关注的是逻辑变更的全局记录；redo log用于确保物理变更的持久性，确保事务能够刷盘成功；undo log是逻辑逆向操作日志，记录的是旧值，方便恢复到事务开始前的状态。

Binlog会记录整个sql或行变化；redo log是为了恢复“已提交但未刷盘”的数据，undo log是为了撤销未提交的事务。

## 7、MVCC了解吗？

MVCC指的是多版本并发控制，每次修改数据时，都会生成一个新的版本，而不是直接在原有数据上进行修改。并且每个事务只能看到它在开始之前 已经提交的数据版本。

其底层实现主要依赖于Undo log和Read View

每次修改数据前，先将记录拷贝到Undo log，并且每条记录会包含三个隐藏列，DB_TRX_ID用来记录修改该行的事务ID，DB_ROLL_PTR用来指定Undo Log中前一个版本，DB_ROW_ID用来标识该行数据。

每次读取数据时，都会生成一个ReadView，其中记录了当前活跃事务的ID集合、最小事务ID、最大事务ID等信息，通过与DB_TRX_ID进行对比，判断当前事务是否可以看到该数据版本。

## 14、Mysql 与Redis一致性保证

MySQL 作为持久化关系型数据库，具备强一致性特性；Redis 作为内存型缓存，具备高性能优势。二者作为异构存储，**强一致性难以实现**（需承担极高的性能损耗），因此绝大多数业务场景均以「最终一致性」为核心目标。以下将对主流一致性保证方案的执行流程、优缺点及适用场景进行详细解析，排序遵循「应用广泛程度+一致性强度」原则。

### 核心背景：一致性问题的根源

1. 读写顺序异常：例如先更新缓存再更新数据库，在并发场景下易产生缓存脏数据；
2. 网络及系统故障：例如数据库更新成功，但缓存删除操作失败（如网络中断、Redis 宕机）；
3. 并发读写冲突：例如读请求加载旧数据至缓存的过程中，写请求完成缓存删除操作；
4. 异构存储缺乏原生分布式事务支持：Redis 作为非关系型数据库，不支持 ACID 事务嵌套，无法与 MySQL 实现原生事务协同。

### 方案 1：Cache Aside（旁路缓存）—— 应用最广泛（最终一致）

#### 核心流程

该方案为业界应用最广泛的一致性保证方案，应用层直接与 MySQL 和 Redis 进行交互，缓存作为数据访问的「旁路」补充。

- 写操作（核心原则：删除缓存而非更新缓存）：先更新 MySQL 数据库，再删除 Redis 缓存（可有效避免并发更新导致的数据覆盖问题）；
- 读操作：优先读取 Redis 缓存，缓存命中则直接返回数据；缓存未命中时，读取 MySQL 数据库，将查询结果写入 Redis 缓存后，再返回给应用层。

#### 关键细节

为何不采用「更新缓存」而选择「删除缓存」？

在并发更新场景中，若存在两个并行的更新请求 A、B，可能出现以下异常时序：

- 请求 A 完成 MySQL 数据更新后，开始执行缓存更新操作（未完成）；
- 请求 B 完成 MySQL 数据更新后，成功执行缓存更新操作；
- 请求 A 继续执行缓存更新，覆盖请求 B 写入的最新缓存数据，最终导致缓存脏数据。

采用删除缓存的方式，后续读请求会从 MySQL 加载最新数据并写入缓存，可有效避免上述缓存脏数据问题。

#### 优点

1. 实现逻辑简洁，无侵入性：无需修改 MySQL、Redis 内核，仅需在应用层实现相关逻辑；
2. 性能均衡：写操作仅增加一次缓存删除步骤，无额外性能损耗；读操作缓存命中时可实现极速响应；
3. 适用性广：适用于 90% 以上的业务场景，包括电商、社交、内容分发等主流领域。

#### 缺点

1. 并发读写一致性不足：若读请求在缓存删除操作执行前已加载旧数据，且在缓存删除后将旧数据写入缓存，会导致缓存脏数据；
2. 缓存删除失败风险：MySQL 数据更新成功后，若因网络中断、Redis 宕机等原因导致缓存删除失败，会造成缓存数据长期与数据库数据不一致；
3. 读穿透风险：当缓存失效或未命中时，所有读请求会直接穿透至 MySQL，可能导致数据库压力过载；
4. 缓存空值问题：当 MySQL 中无对应数据时，若 Redis 缓存空值且未设置过期时间，会导致缓存穿透问题持续存在。

#### 优化手段

- 针对缓存删除失败问题，可通过「MQ 重试机制」「定时任务补偿」两种方式，确保缓存最终被删除；
- 针对缓存空值问题，可为空值缓存设置短期过期时间（例如 5 分钟），避免缓存穿透；
- 针对读穿透问题，可对热点数据设置「永不过期+主动更新」策略，减少缓存未命中场景。

### 方案 2：延迟双删（Cache Aside 优化版）—— 解决并发读写一致性问题

#### 核心流程

该方案针对 Cache Aside 方案中「并发读写导致的缓存脏数据」问题进行优化，在基础缓存删除操作后，增加延迟二次删除步骤，具体流程如下：

更新 MySQL 数据库 → 删除 Redis 缓存 → 延迟 N 毫秒（例如 500 毫秒）→ 再次删除 Redis 缓存。

#### 原理

在并发场景中，读请求可能在第一次缓存删除操作执行前，已从 MySQL 读取旧数据并准备写入 Redis；通过设置延迟时间，可确保读请求完成旧数据写入后，第二次缓存删除操作能及时清除该脏数据，从而保证缓存与数据库数据一致。

#### 优点

1. 低成本解决并发脏数据问题：仅需增加一次延迟删除操作，无需重构核心业务逻辑；
2. 兼容性强：完全兼容原有 Cache Aside 方案的执行逻辑，改造难度较低，成本可控。

#### 缺点

1. 延迟时间难以精准把控：不同业务场景下，读请求从 MySQL 加载数据、写入 Redis 的耗时存在差异，延迟时间过短会导致二次删除无效，延迟时间过长会影响业务性能；
2. 代码复杂度增加：需额外实现延迟任务、异步删除等逻辑，增加应用层开发和维护成本；
3. 二次删除仍存在失败风险：需搭配重试机制，确保二次删除操作最终执行成功。

### 方案 3：Read Through（读穿透）—— 实现应用层解耦

#### 核心流程

该方案中，应用层仅与 Redis 进行交互，Redis 作为数据访问的统一入口，负责与 MySQL 协同工作，具体流程如下：

应用层读取 Redis 缓存 → 缓存未命中时，Redis 主动读取 MySQL 数据库 → 将查询结果写入 Redis 缓存 → 将数据返回给应用层。

#### 优点

1. 解耦性强：应用层无需感知 MySQL 的存在，降低应用层与数据存储层的耦合度；
2. 简化应用层代码：应用层无需单独处理「读数据库、写缓存」的逻辑，由 Redis 统一管控；
3. 缓存策略统一：可在 Redis 侧统一配置缓存过期、数据预热等策略，便于全局管理。

#### 缺点

1. Redis 成为性能瓶颈：所有读请求均需经过 Redis 处理，若 Redis 性能下降，会影响整个读链路的响应速度；
2. 开发成本较高：需自定义 Redis 插件（如 Redis Module）实现缓存加载逻辑，开发和调试难度较大；
3. 首次读取存在延迟：缓存未命中时，Redis 需同步读取 MySQL 并写入缓存，会增加首次读取的响应时间；
4. 可用性依赖 Redis：若 Redis 宕机，整个读链路将无法正常工作，需搭配 Redis 集群及降级策略保障可用性。

### 方案 4：Write Through（写穿透）—— 伪强一致性

#### 核心流程

该方案中，应用层仅与 Redis 进行交互，Redis 作为数据写入的统一入口，同步完成缓存与数据库的更新，具体流程如下：

应用层写入 Redis 缓存 → Redis 同步更新自身缓存数据 → Redis 同步更新 MySQL 数据库 → 向应用层返回写入成功结果。

#### 优点

1. 接近强一致性：Redis 与 MySQL 同步更新，可最大限度保证缓存与数据库数据一致；
2. 应用层解耦：应用层无需关注 MySQL 的更新操作，降低开发复杂度；
3. 数据安全性高：同步写入磁盘（MySQL），可避免数据丢失风险。

#### 缺点

1. 写性能较差：需同步写入 Redis 和 MySQL 两个存储介质，写入耗时约为单一存储的 2 倍，性能损耗显著；
2. Redis 存在单点故障风险：若 Redis 宕机，所有写请求将无法正常执行，影响业务可用性；
3. 侵入性强：需自定义 Redis 写穿透逻辑，对 Redis 本身有一定侵入性；
4. 并发写性能瓶颈：并发写入场景下，Redis 会存在锁竞争问题，进一步降低写操作响应速度。

### 方案 5：Write Back（写回/异步更新）—— 极致写性能

#### 核心流程

该方案以写性能为核心目标，应用层仅写入 Redis 缓存，Redis 异步批量更新至 MySQL 数据库，具体流程如下：

- 异步更新触发时机：Redis 缓存满时、定时任务触发（例如每 10 秒）、缓存数据被淘汰时。

#### 优点

1. 写性能极致：应用层仅写入 Redis 内存，无磁盘 IO 阻塞，可实现高并发写入；
2. 场景适配性强：适用于「写多读少」的业务场景，例如日志采集、实时统计分析等。

#### 缺点

1. 数据丢失风险极高：若 Redis 宕机，未同步至 MySQL 的缓存数据将全部丢失，无法恢复；
2. 一致性最差：属于最终一致性，数据同步延迟可达秒级甚至分钟级，无法满足高一致性需求；
3. 数据冲突难以处理：异步更新过程中，可能出现 MySQL 数据被旧缓存数据覆盖的情况，导致数据不一致。

### 方案 6：分布式事务方案—— 金融级强一致性

针对支付、金融等对数据一致性要求极高的「强一致」场景，需通过分布式事务机制保证 MySQL 与 Redis 的原子性更新，主流方案主要包括以下 3 种：

#### 6.1 2PC（两阶段提交）

- 执行流程：以 MySQL 作为协调者，首先向 Redis 发送「准备请求」；Redis 确认准备完成后，MySQL 执行数据更新操作；随后 MySQL 通知 Redis 执行更新操作；若双方均执行成功，则提交事务；若任一环节失败，则执行回滚操作。
- 优点：理论上可实现数据强一致性，满足金融级核心业务需求；
- 缺点：性能损耗极大，属于阻塞式事务；存在协调者单点故障风险；Redis 对 2PC 协议无原生支持，需自定义实现相关逻辑。

#### 6.2 本地消息表 + 最终一致性

- 执行流程：更新 MySQL 数据时，同步将更新消息写入「本地消息表」，通过 MySQL 事务保证数据更新与消息写入的原子性；通过异步线程轮询本地消息表，向 Redis 发送数据更新或删除请求；若 Redis 操作执行成功，则将消息表中对应消息标记为「完成」；若执行失败，则进行重试（最多重试 N 次），重试失败后将消息转入死信队列，由人工介入处理。
- 优点：可保证数据最终一致性；无单点故障风险；失败场景可通过重试机制兜底，可靠性较高；
- 缺点：存在数据同步延迟（通常为秒级）；本地消息表会增加 MySQL 存储负担；需额外处理幂等性、重试、死信队列等相关逻辑，开发复杂度较高。

#### 6.3 TCC（补偿事务）

- 执行流程：分为三个阶段，Try 阶段（检查 MySQL、Redis 相关资源是否可用）→ Confirm 阶段（执行 MySQL 数据更新及 Redis 缓存删除操作）→ Cancel 阶段（若任一环节失败，执行回滚操作，恢复数据至初始状态）；
- 优点：非阻塞式事务，性能优于 2PC 方案；可灵活适配复杂业务场景；
- 缺点：开发成本极高，需为每个业务场景手写补偿逻辑；需妥善处理网络异常、幂等性等问题，维护难度较大。

### 方案 7：版本号/时间戳方案—— 解决并发更新脏数据

#### 核心流程

该方案通过为每条数据增加「版本号」或「时间戳」字段，实现缓存与数据库数据的版本对齐，Redis 存储「数据+版本号」，MySQL 同步存储该版本信息，具体流程如下：

1. 更新 MySQL 数据时，将对应数据的版本号自增 1；
2. 删除 Redis 缓存时，携带该数据的最新版本号；
3. 读请求从 MySQL 加载数据并写入 Redis 时，仅当加载数据的版本号大于 Redis 缓存中存储的版本号时，才执行写入操作。

#### 优点

1. 可精准解决并发更新导致的缓存脏数据问题，确保缓存数据与数据库数据版本一致；
2. 逻辑清晰，数据版本可追溯，便于问题排查和数据校验。

#### 缺点

1. 存储成本增加：需为每条数据额外存储版本号或时间戳字段，增加数据库和缓存的存储负担；
2. 代码复杂度提升：需额外实现版本号的生成、校验和管理逻辑；
3. 版本号需保证全局递增：需基于 MySQL 自增 ID 等可靠机制生成版本号，避免版本混乱。

### 各方案对比与场景选择

| Cache Aside   | 最终一致   | 高   | 极低 | 90% 以上业务（电商、社交、内容分发等）                 |
| ------------- | ---------- | ---- | ---- | ------------------------------------------------------ |
| 延迟双删      | 最终一致   | 中   | 低   | 高并发读写场景（秒杀、直播等）                         |
| Read Through  | 最终一致   | 中   | 中   | 应用层解耦需求较高的场景                               |
| Write Through | 接近强一致 | 低   | 中   | 数据一致性要求较高的读多写少场景                       |
| Write Back    | 最终一致   | 极高 | 中   | 写多读少、可容忍数据丢失的场景（日志采集、实时统计等） |
| 本地消息表    | 最终一致   | 中   | 中   | 金融、支付等需最终一致性且支持重试的场景               |
| 2PC/TCC       | 强一致     | 极低 | 极高 | 银行、核心交易等对一致性要求极致的场景                 |

### 通用优化建议（适用于所有方案）

1. **缓存过期时间兜底**：为所有缓存数据设置合理的过期时间（例如 10 分钟），即使出现缓存脏数据，也可通过过期机制自动刷新，保障数据最终一致；
2. **重试机制保障**：针对缓存删除、更新失败场景，通过 MQ 重试、定时任务重试等方式，避免单次操作失败导致的数据不一致；
3. **幂等性设计**：确保所有 Redis、MySQL 操作具备幂等性（例如基于唯一 ID 去重），避免重试操作导致的数据重复更新或删除；
4. 缓存降级策略：当 Redis 出现故障时，触发降级机制，应用层直接读取 MySQL 数据库，牺牲部分性能以保障服务可用性；
5. 监控告警机制：实时监控 Redis 缓存命中率、MySQL 读穿透量等关键指标，及时发现数据一致性问题并触发告警，便于快速排查处理。

### 总结

1. **绝大多数业务场景优先选择 Cache Aside + 延迟双删 + 重试机制 的组合方案，可在保证性能的同时，实现数据最终一致性；**
2. 强一致性需求场景（如金融、支付），优先采用 本地消息表 + 最终一致性方案，相较于 2PC、TCC 方案，可在控制开发成本的前提下，满足核心业务一致性要求；
3. 若业务对写性能有极致需求，可选择 Write Back 方案，但需做好数据备份及丢失应急预案，降低数据丢失风险；
4. 缓存过期时间是保障数据最终一致性的「最后一道防线」，所有一致性方案均需搭配该机制使用，避免脏数据长期存在。



## 15、索引下推

InnoDB 二级索引：

- 索引页只存：**索引列 + 主键 ID**
- 想要查 **非索引列**，必须拿着主键 ID 去聚簇索引查整行数据，这个过程就是**回表**
- 回表要读磁盘，**非常慢**
- 

MySQL 分为两层：

- **Server 层**：SQL 解析、优化、逻辑判断
- **存储引擎层**（如 InnoDB）：物理读取索引、数据

#### 1. 无 ICP（传统流程）

1. 引擎根据索引前缀（如 `name`）找到所有候选记录

2. **全部回表**读取完整行

3. 返回 Server 层，再判断其他条件

   -导致大量无效回表

#### 2. 有 ICP（优化后）

1. 引擎遍历索引，**同时判断所有索引内条件**（`name` + `age`）
2. 只对**满足全部条件**的记录回表
3. 返回 Server 层，只回表有效行

```
SELECT * FROM user 
WHERE name LIKE '张%' AND age = 20;
```

**无 ICP**：找到所有姓张的（1000 条）→ 全部回表 → Server 再筛 age=20

**有 ICP**：引擎边扫索引，边筛 `age=20` → 仅 20 条符合 → 仅 20 次回表



如何判断ICP生效？

`EXPLAIN` 查看 `Extra` 列：

- `Using index condition` → **ICP 已开启**

## 2、NIO和IO多路复用是一个东西吗？

1）NIO（非阻塞IO）

- NIO是java提供的一种IO处理方法，属于非阻塞IO模型，基于缓存区（Buffer）和通道操作
- 它允许一个线程处理多个IO操作，通过Selector（选择器）来管理多个通道的状态（如读、写、连接等），从而提高IO操作的效率
- NIO的核心在于事件驱动，通过检查通道是否就绪来决定 是否执行IO操作，避免了阻塞

IO多路复用

IO多路复用是一种操作系统级别的IO模型，允许单个线程监控多个文件描述符(socket)的IO事件（如可读、可写）

常见实现包括select、poll和epoll（linux）

它通过事件通知机制，让线程在多个IO通道中高效切换，而无需为每个通道分配单独的线程。

## 3、进程和线程的区别?

1）定义不同

进程是操作系统资源分配的基本单位，拥有独立的内存空间和资源。

线程是CPU调度的基本单位，同一进程下线程共享内存资源

- 资源隔离

进程间资源彼此独立

同一进程内的线程资源相互共享（如堆、全局变量）

- 开销对比

创建、切换进程的开销较大（涉及上下文切换，内存切换等）

线程开销更小，适合高并发场景。

- 稳定性

一个进程崩溃不影响其他线程

一个线程异常可能影响整个进程

- 通信方式不同

## 4、进程间的通信方式

- 管道
- 消息队列
- 信号、信号量
- 共享内存
- Socket

## 5、线程间的通信方式

- 共享内存 + 锁（synchronized/Lock）
- 条件变量（wait/notify）
- 信号量（Semaphore）
- 阻塞队列（BlockingQueue）
- 原子操作（Atomic Classes）

## 6、了解协程吗 ，为什么协程比线程性能更好？

协程是一种用户态的轻量级线程，它相比传统线程拥有更低的资源开销和更快的调度效率，因此在高并发、IO密集型场景下性能更好。

展开解释：

- 协程是可被挂起和恢复的函数，是一种抢占式用户级线程
- 它与开发者或语言运行时负责调度，而不依赖操作系统调度。

优点：

- 避免内核态上下文切换
- 更轻量的资源占用
- 非阻塞挂起机制
- 调度可控，开销小

# MQ

## 1、Kafka怎么解决重复消费问题

Kafka重复消费的根本原因

Kafka是“至少一次”投递语义，所以：

- 只要消息没被标记为“已提交”offset，Kafka就可能重复投递
- 消费者消费成功但offset提交失败，就可能重复拉取相同消息

解决重复消费的方式：

消费端实现幂等性（本质上，重复消费是允许的，但消费操作要能幂等）

典型方法：

1）消费端实现幂等性：

- 消费携带唯一ID（如UUID/订单号/事务ID）
- 消费前先检查这个ID是否被处理过（如查询Redis或数据库）
- 没处理过才执行，处理后写入“去重记录表”或Redis集合

2）使用幂等性写入目标系统（如幂等性插入数据库）

- 插入操作带有主键或唯一索引（自然实现幂等）
- 消费端事务逻辑封装在数据库事务中（要么全部成功，要么不提交）

## 2、Kafka为什么要使用手动提交offset，它的优点是什么？

手动提交offset能让你在消息处理成功之后再提交offset，从而避免“消息已经消费但offest提交失败”导致的重复消费问题。

Kafka默认是自动提交，周期性提交（如每5秒）

enable.auto.commit = true

auto.commit.interval.ms = 5000

![](images/image25.png)

Kafka 默认使用自动提交 offset，容易导致“消息未处理完但 offset 已提交”的数据丢失问题。使用手动提交 offset，可以在消费处理完成之后再安全地提交 offset，从而确保消息处理的可靠性和数据一致性，是处理关键业务、金融级场景下的最佳实践。

## 3、Kafka手动提交三种方式：

手动提交 ，当enable.auto.commit被设置为false可以有以下三种提交方式

- 提交当前偏移量（同步提交）
- 异步提交
- 同步和异步组合提交

## 1、HTTP是明文传输吗？HTTPS用什么方式保证了数据加密？讲一下怎么进行CA证书校验

HTTP是明文传输

HTTPS通过TLS/SSL协议保证数据加密，结合非对称加密和对称加密实现安全通信。

加密方式：

## 3、TCP如何解决粘包？

TCP是一种面向流的协议，数据以字节流形式传输，没有明确的边界。这可能导致粘包（多个数据包粘在一起）或半包（一个数据包被拆分）问题。

- 固定长度消息

每个数据包使用固定长度，接收方按照固定长度读取数据

- 分隔符分包

在每个数据包末尾添加特定分隔符（如\n,\r\n 或自定义字符），接收方通过分隔符识别数据包边界。

- 长度前缀

在每个数据包前添加长度字段（通常为固定字节数，如4字节整数），表示数据包的长度。接收方先读取长度字段，再根据长度读取完整数据。

​		如果想要实现这个算法的话也很简单，准备一个队列来保存请求，然后我们定期从队列中拿请求来执行

**分布式限流**部分，在微服务多实例部署场景下的解决方案：

## ES

![img](./assets/041adf6d5f78929349c855838adc9f69.png)

- #### es的架构


es是一个分布式的搜索和分析引擎，它能够存储、搜索和分析大量数据。为了实现这些功能，Elasticsearch采用了分片和副本的机制，使得数据可以分布在多个节点上， 并且具有容错性和可拓展性。

**节点与集群**：Elasticsearch由多个节点（Node）组成，这些节点可以组成一个集群（cluster）。每个节点都可以处理读写请求，并且数据在集群中的节点之间进行分布与复制，以实现高可用性和拓展性。

**分布与副本**：为了支持大规模数据，Elasticsearch将**索引**划分为多个分片（Shared），每个分片可以独立存储和处理数据。此外，每个分片可以有多个副本（Replica），用于提供数据冗余， 故障恢复和读写负载均衡。



- #### 索引与搜索


倒排索引：Elasticsearch使用Lucene作为其底层的搜索库。Lucene构建倒排索引 来加速搜索过程。倒排索引 将文档中单词映射到包含这些单词的文档列表，从而实现快速查找和效率。

查询执行：当客户端发送搜索请求时，请求首先达到一个协调节点。 协调节点解析查询语句，确实需要访问的分片，并将请求转发给相应的数据节点。数据节点在本地执行查询，并将结果给协调节点。协调节点聚合来自各个数据节点的结果，并进行排序、分页等处理，并将结果返回给客户端。

- #### 数据写入与持久化


写入流程：当文档被写入Elasticsearch时，它们首先被放置在内存中的一个缓存区内，并同时记录到事务日志中以确保数据的持久性。随着时间的推移或达到一定的条件，缓冲区中的数据被刷新到Lucene的索引中，形成新的段。这些段是不可变的，一旦写入就不能被修改。最后，通过flush操作将内存中的数据， 以及Translog中的更改持久到磁盘中。

- #### 缓存与性能优化


查询缓存：Elasticsearch会对查询结果进行缓存 以加速重复查询的响应速度。此外，还有分片请求缓存用于减少不必要的计算和IO操作

优化策略：为了提高性能，Elasticsearch还提供了多种优化策略，如使用合适的分析器（Analyzer）和查询语句、合理配置索引设置、利用聚合和过滤操作等。

## 1、es的数据写入与原理

![img](./assets/42c1db09ba730606783114d7eb2f5ee0.png)

  1）客户端想es集群发送一个写入请求，这个请求可以发送到**集群中任何一个节点**。

​	接收到请求的节点会充当协调节点的角色。协调节点负责处理客户端的请求，并将请求路由到正确的数据节点。
2. 路由与主分片处理
    **协调节点**会根据**文档的_id和索引的设置**（如分片数量）来确定文档应该写入到哪个主分片。这是通过一个哈**希函数和模运算**来实现的，**确保同一个_id的文档总是路由到同一个主分片。**
    确定目标主分片后，协调节点将请求转发给该主分片所在的数据节点。
    数据节点上的主分片接收到请求后，会先将文档**写入到内存中的Lucene索引结构**里。这个过程包括将文档**转换成倒排索引**的形式，以便后续的搜索和分析。
3. 数据同步与副本分片
    一旦文档被写入到主分片，主分片会开始将**数据同步到其对应的副本分片**上。这是为了保证数据的冗余和可用性。
    副本分片是主分片的完整拷贝，它们可以处理搜索请求并提供数据恢复的能力。当主分片不可用时，副本分片可以被提升为新的主分片。
    **数据同步**是**异步进行**的，这意味着写入请求在主分片处理完毕后就可以返回给客户端，而不需要等待所有副本分片都完成同步。
4. 写入确认与响应
    当主分片和足够数量的副本分片（根据配置可能是全部或大多数）都成功写入了文档后，**协调节点会收到这些分片的确认信息**。
    一旦收到足够的确认信息，协调节点就会向客户端发送一个**成功的响应**，表示文档已经被成功写入。
5. 底层写入机制
    在Elasticsearch中，**底层写入机制是确保数据可靠、持久化并可以被高效搜索的关键部分**

## 2、**es读取数据流程**

![img](./assets/9d3c11c2eee724343009f3197d5ee823.png)

##### 1. 客户端发送请求

- 当用户想要从Elasticsearch中检索数据时，他们会通过Elasticsearch的客户端API发送一个搜索请求。这个请求包含了查询的详细信息，如要**搜索的索引、查询类型（如匹配查询、范围查询等）、过滤条件**等。

##### 2. 请求到达协调节点

- 请求首先到达Elasticsearch集群中的一个节点，这个节点被称为**协调节点**。协调节点负责接收客户端的请求，处理请求的路由逻辑，并与**数据节点**进行通信以获取实际的数据。

##### 3. 解析查询并确定目标分片

- 协调节点接收到请求后，会解析查询语句，并根据索引的映射（Mapping）和设置（Settings）信息来确定需要查询哪些分片（Shard）。Elasticsearch中的每个索引都被分割成多个分片，并且这些分片可以分布在集群的多个节点上以提高可扩展性和性能。

##### 4. 将请求转发给数据节点

- 协调节点根据分片的位置信息将查询请求转发给包含目标分片的数据节点。**每个数据节点上都存储着一部分索引的数据**，并负责处理与这些数据相关的查询请求。

##### 5. 在数据节点上执行查询

- 数据节点接收到查询请求后，会使用**Lucene库**来执行实际的搜索操作。Lucene是一个高性能、全功能的文本搜索引擎库，它提供了强大的索引和搜索功能。数据节点会**根据查询条件在Lucene索引中检索匹配的文档**，并生成一个结果集。

##### 6.聚合和排序结果

- 数据节点将查询结果返回给协调节点。如果查询涉及多个分片，协调节点需要**聚合**来自不同分片的结果，并根据需要对结果进行**排序、分页**等处理。这个过程可能需要**消耗一定的计算资源，特别是当结果集很大时**。

##### 7.返回结果给客户端

- 一旦结果准备好，协调节点会将它们**封装成一个统一的响应格式**，并返回给客户端。响应中包含了查询的结果、匹配的文档数量、聚合数据（如果有的话）等信息。客户端可以解析这个响应来获取所需的数据。

- 注册分支事物的过程

    ↓
  handleTextMessage() 方法

  2. **WebSocketConfig 的路由配置**

  看 WebSocketConfig.java 第 20 行：
  registry.addHandler(chatWebSocketHandler, "/chat/{token}")

  - 路径：/chat/{token}
  - 处理器：ChatWebSocketHandler
  - 结果：所有发往 /chat/{token} 的 WebSocket 连接都由 ChatWebSocketHandler 处理

  注意：这里没有提到 ChatController！

**这是什么设计模式？**

  1. 策略模式（Strategy Pattern） ⭐

  WebSocketConfig 和 ChatWebSocketHandler 的关系是典型的策略模式：

  // WebSocketConfig.java
  registry.addHandler(chatWebSocketHandler, "/chat/{token}")

  核心思想：将不同的处理逻辑封装成独立的策略类，运行时动态选择。

  WebSocketHandlerRegistry（上下文）
    ↓
  注册多个 Handler（策略）
    ├─ ChatWebSocketHandler（聊天策略）
    ├─ NotificationHandler（通知策略，如果有）
    └─ FileTransferHandler（文件传输策略，如果有）

  好处：
  - 可以为不同路径注册不同的 Handler
  - 每个 Handler 独立实现自己的逻辑
  - 易于扩展新的 WebSocket 功能

  例子：
  @Override
  public void registerWebSocketHandlers(WebSocketHandlerRegistry registry) {
      // 策略1：聊天处理
      registry.addHandler(chatWebSocketHandler, "/chat/{token}");

      // 策略2：通知处理（假设有）
      registry.addHandler(notificationHandler, "/notifications/{userId}");
    
      // 策略3：实时协作（假设有）
      registry.addHandler(collaborationHandler, "/collab/{roomId}");
  }

---
  2. 模板方法模式（Template Method Pattern） ⭐

  ChatWebSocketHandler 继承 TextWebSocketHandler：

  public class ChatWebSocketHandler extends TextWebSocketHandler {
      @Override
      protected void handleTextMessage(WebSocketSession session, TextMessage message) {
          // 具体实现
      }

      @Override
      public void afterConnectionEstablished(WebSocketSession session) {
          // 具体实现
      }
  }

  核心思想：父类定义算法骨架，子类实现具体步骤。

  TextWebSocketHandler（抽象模板）
    ├─ handleMessage()（模板方法，定义流程）
    │    ├─ 1. 检查消息类型
    │    ├─ 2. 调用 handleTextMessage()（钩子方法）
    │    └─ 3. 异常处理
    │
    └─ ChatWebSocketHandler（具体实现）
         └─ 重写 handleTextMessage()（填充具体逻辑）

  Spring 的 WebSocket 处理流程（模板方法）：
  // TextWebSocketHandler 内部（简化版）
  public final void handleMessage(WebSocketSession session, WebSocketMessage<?> message) {
      if (message instanceof TextMessage) {
          handleTextMessage(session, (TextMessage) message);  // 调用子类实现
      } else if (message instanceof BinaryMessage) {
          handleBinaryMessage(session, (BinaryMessage) message);
      }
  }

---
  3. 注册表模式（Registry Pattern）

  WebSocketHandlerRegistry 本身就是注册表模式：

  registry.addHandler(handler, path)

  核心思想：维护一个映射表，根据 key（路径）查找 value（Handler）。

  WebSocketHandlerRegistry（注册表）
    ├─ "/chat/{token}" → ChatWebSocketHandler
    ├─ "/notifications/{userId}" → NotificationHandler
    └─ "/collab/{roomId}" → CollaborationHandler

  运行时查找：
  客户端连接 ws://localhost:8081/chat/abc123
    ↓
  注册表查找："/chat/{token}" 匹配成功
    ↓
  返回：ChatWebSocketHandler
    ↓
  调用：chatWebSocketHandler.handleTextMessage()

---
  4. 依赖注入（Dependency Injection）

  @Configuration
  public class WebSocketConfig implements WebSocketConfigurer {
      @Autowired
      private ChatWebSocketHandler chatWebSocketHandler;  // Spring 注入
  }

  核心思想：不自己创建依赖对象，由容器注入。

  好处：
  - 解耦：Config 不需要知道 Handler 的创建细节
  - 可测试：可以注入 Mock 对象
  - 单例管理：Spring 自动管理生命周期

---
  5. 配置模式（Configuration Pattern）

  @Configuration
  @EnableWebSocket
  public class WebSocketConfig implements WebSocketConfigurer {
      // 集中配置 WebSocket 相关设置
  }

  核心思想：将配置逻辑集中在一个地方，与业务逻辑分离。

  类比：
  - WebSocketConfig = 路由配置文件（类似 Spring MVC 的 @RequestMapping）
  - ChatWebSocketHandler = 业务逻辑处理器（类似 @RestController）

---
  完整的设计模式组合

  ┌─────────────────────────────────────────────┐
  │  WebSocketConfig（配置模式 + 注册表模式）      │
  │  ┌───────────────────────────────────────┐  │
  │  │ @Configuration                        │  │
  │  │ @EnableWebSocket                      │  │
  │  │                                       │  │
  │  │ @Autowired                            │  │
  │  │ ChatWebSocketHandler handler ←─────────┼──┐ 依赖注入
  │  │                                       │  │  │
  │  │ registerWebSocketHandlers() {        │  │  │
  │  │   registry.addHandler(handler, path) │  │  │
  │  │ }                                     │  │  │
  │  └───────────────────────────────────────┘  │  │
  └─────────────────────────────────────────────┘  │
                      ↓ 注册                        │
  ┌─────────────────────────────────────────────┐  │
  │  WebSocketHandlerRegistry（注册表）          │  │
  │  ┌───────────────────────────────────────┐  │  │
  │  │ Map<String, WebSocketHandler>         │  │  │
  │  │   "/chat/{token}" → handler           │  │  │
  │  └───────────────────────────────────────┘  │  │
  └─────────────────────────────────────────────┘  │
                      ↓ 路由                        │
  ┌─────────────────────────────────────────────┐  │
  │  TextWebSocketHandler（模板方法模式）        │  │
  │  ┌───────────────────────────────────────┐  │  │
  │  │ handleMessage() {                     │  │  │
  │  │   if (TextMessage)                    │  │  │
  │  │     handleTextMessage() ←─────────────┼──┼──┘ 钩子方法
  │  │ }                                     │  │
  │  └───────────────────────────────────────┘  │
  └─────────────────────────────────────────────┘
                      ↑ 继承
  ┌─────────────────────────────────────────────┐
  │  ChatWebSocketHandler（策略模式）            │
  │  ┌───────────────────────────────────────┐  │
  │  │ @Component                            │  │
  │  │                                       │  │
  │  │ @Override                             │  │
  │  │ handleTextMessage() {                 │  │
  │  │   // 具体的聊天处理逻辑               │  │
  │  │   chatHandler.processMessage(...)     │  │
  │  │ }                                     │  │
  │  └───────────────────────────────────────┘  │
  └─────────────────────────────────────────────┘

---
  类比理解

  HTTP Controller（你熟悉的）

  @RestController
  @RequestMapping("/api/v1/users")
  public class UserController {
      @GetMapping("/{id}")
      public User getUser(@PathVariable Long id) { ... }
  }

  等价于：

  WebSocket Handler

  @Configuration
  public class WebSocketConfig {
      registry.addHandler(chatHandler, "/chat/{token}");
  }

  @Component
  public class ChatWebSocketHandler extends TextWebSocketHandler {
      @Override
      protected void handleTextMessage(...) { ... }
  }

  对比：

  ┌────────────────────────┬───────────────────────────────┐
  │          HTTP          │           WebSocket           │
  ├────────────────────────┼───────────────────────────────┤
  │ @RequestMapping        │ registry.addHandler()         │
  ├────────────────────────┼───────────────────────────────┤
  │ @RestController        │ extends TextWebSocketHandler  │
  ├────────────────────────┼───────────────────────────────┤
  │ @GetMapping            │ @Override handleTextMessage() │
  ├────────────────────────┼───────────────────────────────┤
  │ DispatcherServlet 路由 │ WebSocketHandlerRegistry 路由 │
  └────────────────────────┴───────────────────────────────┘

---
  总结

  这个设计使用了：

  1. 策略模式：不同路径使用不同 Handler
  2. 模板方法模式：父类定义流程，子类实现细节
  3. 注册表模式：路径到 Handler 的映射
  4. 依赖注入：Spring 管理对象生命周期
  5. 配置模式：集中配置，分离关注点

  核心优势：
  - ✅ 解耦：配置、路由、业务逻辑分离
  - ✅ 扩展：新增 WebSocket 功能只需添加新 Handler
  - ✅ 复用：多个路径可以共享同一个 Handler
  - ✅ 可测试：每个组件可以独立测试

  这是 Spring 框架的经典设计，和 Spring MVC 的 Controller 机制非常相似！

```java
    void solve( int i,int[] nums,List<List<Integer>>ans,Boolean[] flag,int n,List<Integer>list){

            ans.add(new ArrayList<>(list));
            return ;
        }

        for(int j=0;j<n;j++){
            if(flag[j] == false){
                flag[j]=true;
                list.add(nums[j]);
                solve(j+1,nums, ans ,flag ,n,list);
                list.remove(list.size()-1);
                flag[j]=false;
            }
        }

    public List<List<Integer>> permute(int[] nums) {

        List<List<Integer>> ans=new ArrayList<>();
        List<Integer> list=new ArrayList<>();
        int n=nums.length;
        Boolean []flag = new Boolean[n] ;
        for(int i=0;i<n;i++)flag[i]=false;
        solve(0,nums,ans ,flag,n ,list);
        return ans;

    }
}
```

```java
如果写 ans.add(list);
```
